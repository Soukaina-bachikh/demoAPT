# CLAUDE.md

This file documents the **demoAPT** project for anyone (human or AI) working on it next. It reflects what was actually built and debugged, not just the original spec — several sections exist specifically because something surprising was discovered the hard way.

## Project Overview

**demoAPT** is a 4D v21 application demonstrating AI-powered appointment booking using **OpenAI function calling (tools)** via the **4D AIKit** component. The AI decides which 4D method to call, the method executes DB logic, and the result feeds back into the model for a natural-language reply.

- **Client interface**: `Chat` — a native 4D form (title bar, message input, Send button) wrapping a Web Area used purely as a bubble-rendering surface.
- **Staff interface**: `Admin` — a native 4D form for searching, filtering, and cancelling appointments.
- **AI model**: `gpt-4o` via `cs.AIKit.OpenAI`, called with **direct streaming `chat.completions.create()`**, not AIKit's higher-level `chat.create()`/`registerTools()` helper — tool dispatch is handled manually in this project, on purpose (see Architecture below).
- **Language**: English throughout.

---

## First-time setup

1. Open `demoAPT.4DProject` in 4D v21.
2. Let the Component Manager download **4D AIKit** (the only hard dependency).
3. Copy `Resources/AIProvider.example.json` to `Resources/AIProvider.json` and put your real OpenAI API key in it. **This file is gitignored — never commit it.**
4. Run `APT_Seed` once from the Method Editor to populate sample data (safe to re-run — it clears existing data after a confirmation prompt).
5. Run `APT_Open("")` for the client chat window, or `APT_Open("admin")` for the staff panel.

## Data model (5 tables)

All primary keys are UUID text, **auto-generated by ORDA on `.new()`** — never assign them manually (see gotchas). Foreign keys are plain text fields, matched by manual queries — there are no ORDA relations defined in the catalog.

| Table | Primary Key | Key Fields |
|---|---|---|
| `Staff` | `staffID` | firstName, lastName, specialty, slotDuration (minutes), isActive |
| `Client` | `clientID` | firstName, lastName, email, phone, createdAt |
| `Appointment` | `appointmentID` | confirmationCode (`APT-XXXXXX`), clientID, staffID, date, time, duration, reason, status (`confirmed`/`cancelled`) |
| `Availability` | `availabilityID` | staffID, dayOfWeek (1=Mon...7=Sun, ISO convention), startTime, endTime |
| `Conversation` | `conversationID` | appointmentID, clientID (both linked once resolved — see Chat Architecture), messages (JSON text), startedAt, closedAt (text timestamps, not native Date/Time) |

Field type reference used in `catalog.4DCatalog`: `10`=Text, `4`=Long, `1`=Boolean, `8`=Date, `9`=Time/Duration. There is **no native timestamp type** — `Conversation.startedAt`/`closedAt` are stored as plain ISO-ish text (`"yyyy-MM-dd"+"T"+"HH:mm:ss"`), not real Date/Time fields.

## AI architecture: OpenAI tool calling

### Flow

```
User types (native input) → btnSend/messageInput → APT_Chat_SubmitInput
  → APT_Chat_Receive($text)
      → pushes user bubble into web area via WA EXECUTE JAVASCRIPT FUNCTION
      → APT_Chat_Send($text)
          → $openAI.chat.completions.create(...) with 12 tool definitions, stream:True
              → APT_Chat_OnData($chunk)   → accumulates content AND tool_calls deltas
              → APT_Chat_OnTerminate($result)
                  if finish_reason = "tool_calls":
                      → APT_Tool_Dispatch(name, args) → APT_Tool_*(args)
                      → track clientID/appointmentID on the Conversation (see below)
                      → append tool result to messages
                      → streamStart (reuses existing placeholder bubble, doesn't orphan it)
                      → APT_Chat_ContinueStream(messages)  → second streaming pass
                  else:
                      → streamEnd(fullReply) → parses <slots> markers into buttons
                      → save Conversation.messages/clientID/appointmentID
```

### Streaming tool_calls has to be manually accumulated

OpenAI streams `tool_calls` fragmented across multiple chunks, tagged by `index` — `id`/`type`/`function.name` arrive once, `function.arguments` arrives token-by-token and must be concatenated. **`$choice.message.tool_calls` is never populated in the streaming terminate result** — only `$chunk.choice.delta.tool_calls` fragments during `OnData`. `Form.toolCallsBuffer` (reset per turn in `APT_Chat_ResetChunk`) accumulates these; `OnTerminate` reads from that buffer, never from `$choice.message.tool_calls`.

### The 12 tools

| Tool | Purpose |
|---|---|
| `checkAvailability` | Free slots for a date (optional staffID filter) |
| `getStaffList` | Active staff (optional specialty filter) |
| `getNextAvailable` | Next free slot for one staff member from a date |
| `getWeekAvailability` | Free slots across the next 7 days for one staff member — lets the model proactively suggest alternatives instead of dead-ending when a specific date is full |
| `createAppointment` | Books a slot, generates `APT-XXXXXX` code |
| `cancelAppointment` | Sets status to `cancelled` |
| `rescheduleAppointment` | Moves to new date/time after a slot check |
| `listAppointments` | Client's appointments (upcoming/past/all) |
| `getAppointmentDetails` | Full detail by internal UUID |
| `getAppointmentByCode` | Lookup by `APT-XXXXXX` confirmation code |
| `findClient` | Search by name/email/phone — **always called before createAppointment** |
| `createClient` | Register new client — defensively re-checks for a duplicate itself (see below), not just relying on the model calling `findClient` first |

Tool schemas are built in `APT_GetToolDefinitions` via `cs.AIKit.OpenAITool.new({name; description; parameters})` (the simplified format — no `handler:` key, since this project uses the manual-dispatch pattern, not AIKit's tool-calling helper).

### Conversation linking

`Conversation.clientID`/`appointmentID` are populated as a side effect of tool dispatch in `APT_Chat_OnTerminate`: whenever a tool result contains a `.client` object (from `findClient`/`createClient`) or `createAppointment` succeeds, `Form.currentClientID`/`currentAppointmentID` are updated and persisted to the Conversation record the next time it's saved.

### The `<slots>` marker → clickable buttons

Rule 7 of `APT_SystemPrompt` instructs the model to append a machine-readable block after presenting concrete bookable options: `<slots>{"staffName":"...","options":[{"date":"2026-07-21","time":"14:30"}]}</slots>`. `chat.html`'s `formatReply()` strips this from the visible text and renders it as clickable pill buttons. Clicking one sends `"Book me for Tue, Jul 21 at 14:30 with Dr. X."` through `window.$4d.APT_Chat_Receive(...)` — the **only** JS→4D bridge call in the whole app (see Chat Form Architecture).

## Chat form architecture

`Chat` is a **native form**, not a browser page pretending to be one. The Web Area only renders message bubbles — it has no header, no composer, no `$4d` bridge calls except the one slot-picker click handler.

- Native `messageInput` (bound to `Form.messageText`) + native `btnSend` → `APT_Chat_SubmitInput` → `APT_Chat_Receive`.
- 4D pushes into the page via `WA EXECUTE JAVASCRIPT FUNCTION(*; "webArea"; "streamChunk"/"streamEnd"/"streamStart"/"streamError"/"addUserBubble"/"resetMessages"; ...)`.
- The Web Area requires **`"methodsAccessibility": "all"`** and **`"webEngine": "embedded"`** set explicitly in `form.4DForm` — without `webEngine: embedded`, it can default to a legacy engine that doesn't support `Element.closest()` (the click handler is written without `.closest()`/`NodeList.forEach` anyway, for safety).
- The web inspector is enabled on load (`WA SET PREFERENCE(*; "webArea"; WA enable Web inspector; True)`) — right-click → Inspect to debug the JS side.

## Admin form architecture

`Admin` is a plain native form: search field + filter buttons (Upcoming/Past/All/Cancelled) + a `listboxType: "collection"` bound to `Form.appointments`, with a detail panel bound via `currentItemSource: "Form.currentAppointment"`.

- **Listbox column `dataSource` must be `This.propertyName`** (e.g. `This.confirmationCode`), not `Form.appointments.confirmationCode` and not `Form.appointments[].confirmationCode` — both of those silently render blank cells. `This` refers to the current row within the collection iteration. `currentItemSource` binds fine with a plain `Form.xxx` path since it's a single object, not a per-row iteration.
- Detail-panel fields bound to `Form.currentAppointment.xxx` must be `type: "input"` with `enterable`/`focusable: false`, **not** `type: "text"` — plain text objects only support a static literal `text` string, not a `dataSource` at all.

## Naming conventions

- **Method prefix**: `APT_` for application methods, `AdminForm_` for Admin-panel-specific helpers.
- **Tool handlers**: `APT_Tool_` prefix, one per tool, dispatched from `APT_Tool_Dispatch`.
- **Confirmation codes**: `APT-XXXXXX` — 6 uppercase alphanumeric chars from `Uppercase(Substring(Generate UUID; 1; 6))`.
- **Table names**: singular (`Staff`, not `Staffs`).

## Key rules (enforced via system prompt, `APT_SystemPrompt`)

1. `findClient` before `createAppointment` — never book without a resolved `clientID`.
2. Never show UUIDs to the user — only the `APT-XXXXXX` code.
3. Mentioning a confirmation code → call `getAppointmentByCode` immediately.
4. Always verify availability before booking/rescheduling.
5. `createClient` only when `findClient` found no match (also defensively enforced in code — see below).
6. A specific date/time that's unavailable → immediately call `getWeekAvailability` and offer alternatives instead of just reporting failure.
7. Present concrete options via the `<slots>` marker (max 6), keep prose short.
8. Today's date is injected fresh into the prompt every turn (`Current date` + weekday name) — the model has no other way to know "today," and will otherwise guess/hallucinate a date.

## 4D gotchas discovered while building this (read before "fixing" something that isn't broken)

These cost real debugging time. Don't reintroduce them.

- **`GetOption("...")` does not exist.** There is no generic option getter/setter in 4D (only namespaced ones like `SET PRINT OPTION`). API keys are read from `Resources/AIProvider.json` via `JSON Parse` + `Folder(fk resources folder)` instead (`APT_GetOpenAIKey`).
- **`Day of` returns the day of the *month* (1-31), not the weekday.** Use `Day number` (1=Sunday...7=Saturday), then convert to this project's ISO convention (1=Monday...7=Sunday) with `dow:=Day number(date)-1; If (dow=0) dow:=7`.
- **`String(Null)` returns the literal text `"null"`**, not an empty string. Always guard: check `#Null` before calling `String()` on anything that might be absent (see `APT_TextOrEmpty`, and the `delta.content` guard in `APT_Chat_OnData`). This caused visible "null" bubbles in the chat and would have silently written the word "null" into optional fields like `reason`/`email`/`phone`.
- **`String(date)` needs an explicit literal pattern string**, e.g. `String($date; "yyyy-MM-dd")` / `String($time; "HH:mm")` — there's no bare `ISO date` constant that works. Confirmed from real working 4D code (a 4d-depot example), not from official docs.
- **ORDA's `.query()` string mini-language does not support inline field-concatenation expressions** like `(firstName+' '+lastName) = :1`. That clause silently never matches anything — it doesn't error, it just never fires. This caused `findClient` to always report "not found" for full names, creating a duplicate client on every request. Because query-string matching semantics have proven unreliable more than once here, client matching (`APT_FindClientEntity`) is done with a plain 4D-code loop and explicit `Uppercase()` comparisons instead of trusting `.query()`'s comparison behavior.
- **Primary keys are auto-generated by ORDA on `.new()`.** Don't call `Generate UUID` and assign it yourself for a table's own primary key — read it back after `.new()`/`.save()` instead if you need it for a foreign key elsewhere (see `APT_Seed`). `Generate UUID` is still legitimately used for building the `APT-XXXXXX` confirmation code substring, which is a unique *field*, not the primary key.
- **`WA OPEN URL` / `WA EXECUTE JAVASCRIPT FUNCTION` take the web area as a quoted string name with `*` as the first argument** — `WA OPEN URL(*; "webArea"; url)`, `WA EXECUTE JAVASCRIPT FUNCTION(*; "webArea"; "fnName"; $resultVar; ...args)` — not a bare object reference or pointer, and the JS-function variant needs an output variable slot before the actual arguments. Confirmed from a real working 4d-depot example.
- **`"formClass"` in `form.4DForm` is a real schema key but crashed this project's Form Editor.** It's documented in `formsSchema.json` and used successfully in a real 4d-depot example, but reintroducing it here reliably froze/crashed the app. Both forms use the classic plain-`Form`-object pattern instead. If you're tempted to add it back, test on a throwaway form first.
- **A form's Web Area defaults to a legacy rendering engine** unless `"webEngine": "embedded"` is set explicitly — the legacy engine may not support modern JS (`Element.closest()`, `NodeList.forEach`). Always set it explicitly, and write JS defensively regardless.
- **Hand-authoring `.4DForm` JSON is risky in general.** Several real bugs here only got found by having 4D itself generate/resave a form and diffing against what was hand-written (blank object structure, the `This.property` listbox column convention, etc.). When something in a form doesn't work and the fix isn't obvious, building a minimal version by hand in the actual Form Editor and inspecting the result is more reliable than guessing at JSON.
